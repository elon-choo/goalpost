# /goalpost:run — hands-off autonomous execution from the ledger

Enter the **goalpost** skill's RUN mode. `$ARGUMENTS` may name a specific ledger path and/or `--stage N` to bound the run; if empty, auto-select the newest ledger.

Follow the skill's RUN procedure exactly:
1. Run the preflight; confirm the `mcp__codex__codex` tool is actually available (if not, declare the Claude-only degraded mode).
2. Fix the target repo first (user-named path, else `git rev-parse --show-toplevel`, else cwd — **but if rev-parse failed and the cwd is not a git repo while containing multiple projects, e.g. `~` or `~/Documents`, the target repo is UNRESOLVED: do not sweep the subtree, ask the user**), then auto-select the active ledger among goalpost-generated `LEDGER*.md` (`Generator: goalpost`) **inside that repo only** — never from other projects, shared doc folders, recall, or memory; **no ledger in this repo → stop and ask, don't widen the search.** Pick newest by `Created:` header (tie-break filename timestamp → mtime); integrity-check it (has `▶ NEXT`, status codes, ≥1 goal, and a `Target repo:` matching the current repo — mismatch means a foreign/mirror ledger: stop and ask) and claim it with a `Session-claim` line — if a fresh claim by another session exists, report the conflict and stop (**never kill the other session**). Declare path + completion % + stop conditions + budget in one line. If `▶ NEXT` is a `[~]` goal, use the claim/`Last updated` timestamp + a repo scan for partial artifacts as the backstop (treat liveness as opportunistic, not guaranteed): fresh → wait; stale → re-fire without a strike after the artifact scan.
3. If the host supports it, set a heartbeat (ScheduleWakeup safety net; offer CronCreate that resumes only on a stale claim). If ScheduleWakeup/CronCreate are unavailable, say so and run without a heartbeat.
4. Loop, one goal at a time (or several independent goals concurrently — see below): re-read ledger + refresh claim → check budget → fire the goal under the goal-execution contract (`${CLAUDE_PLUGIN_ROOT}/commands/goal.md`) at its `[<platform>:<model>]` tier (Codex `model: "gpt-5.6-<tier>"` + effort + the guardrail `developer-instructions`; a `[fanout]` goal → one Workflow stage-runner, one Codex call per sub-task at its mapped tier) → verify the DoD with first-party evidence (distinguish DoD-fail / infra-error / flake) → update the ledger → move the pointer (only when its `depends:` are all `[x]`). **A DoD strike escalates one model tier** (Luna→Terra→Sol; Terra→Terra→Sol) rather than repeating; `quality_strikes` cap 3 → HUMAN_GATE; `infra_retries` are separate (don't count/escalate); a **SAFETY_STOP** (worker refused a forbidden action) is neither — it's `HUMAN_GATE(safety-stop)`, never escalated to a stronger tier. A HIGH-blast-radius goal whose required tier (Sol) is unavailable → `HUMAN_GATE(model-unavailable)`, never a Terra downgrade; a bare `[codex]` is unclassified — classify before dispatch, don't default a security goal to Terra. **Throughput (wave scheduler):** goals whose `depends:` are `[x]` **and whose declared write sets are disjoint** may run concurrently (bounded) at their own tiers; overlapping/undeclared write sets serialize or worktree-isolate; a failed leaf retries alone; each still gets its own first-party gate before `[x]`. A **destructive-capable** goal (direct or transitive — e.g. a test whose hook touches a real DB) runs isolated (`workspace-write`, no ambient prod secrets), and actual data destruction is a HUMAN_GATE.
5. At each stage boundary run in order: `G x.9` integration + review GO → `G x.10` generate next-stage detail → `G x.9.5` independent transition review (vets the generated DoDs). A NO-GO, a weakening DoD change, `codex-down`, `review-unavailable`, or budget reached is a HUMAN_GATE.
6. Report only at a stage completion, a HUMAN_GATE, or full completion — not per goal.

Unless the user said "to the end" or named a stage count, honour `--stage N` or stop at the first HUMAN_GATE and report.
